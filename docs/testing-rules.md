# 系统级前后端深度验证规则

> 本规则不是某个项目的测试清单，而是所有软件系统交付前的系统级验证框架。
> 核心目标：从第一性原理判断——这个系统是否真的能完成它声称要完成的目标，且前后端实现是否可靠、可维护、可扩展、可迭代、可交付。

---

## 0. 根原则

### 0.1 先问系统目标，再看代码和页面

任何测试开始前，必须先回答：

1. 这个系统服务谁？
2. 用户要用它完成什么任务？
3. 系统的核心业务闭环是什么？
4. 哪些能力是 MVP 必须真实可用的？
5. 哪些只是说明、配置入口、未来能力或 mock？
6. 如果这个系统今天交给真实用户，用户第一天会怎么用？会在哪些地方卡死？

如果没有回答这些问题，直接跑 build、点页面、列接口，都属于无目标测试。

### 0.2 技术通过不等于系统可用

- `tsc --noEmit` 通过：只能说明类型检查暂时没爆
- `npm run build` 通过：只能说明能构建
- 页面 HTTP 200：只能说明路由可达
- API 返回 200：只能说明接口有响应
- 单元测试全绿：只能说明被测的边界情况能过

这些都不能证明系统能完成业务目标。

### 0.3 验证顺序

必须按以下顺序验证，每阶段有硬门禁：

1. **目标验证**：系统目标、用户路径、业务闭环是否成立
2. **架构验证**：前后端语言、框架、模块边界、数据流、安全性是否合理
3. **后端验证**：API、数据库、服务、任务流、并发、缓存、安全、性能
4. **前端验证**：功能、接口、状态、交互、URL 状态、表单、UI、性能、可访问性
5. **联调验证**：前后端契约、错误处理、真实数据流、认证全链路、权限边界
6. **交付验证**：回归、文档、可运维性、风险清单、未完成边界

跳过前面阶段直接测页面，属于低质量验收。

## 1. 系统目标与业务闭环验证

### 1.1 系统定位检查

必须先写出系统定位：

```text
系统名称：
目标用户：
核心任务：
输入是什么：
输出是什么：
关键业务闭环：
真实交付物：
不可接受的失败：
```

示例：

```text
目标用户：漫剧生产团队
核心任务：从故事设定 → 分镜 → 资产 → 渲染 → QA → 剪辑 → 交付
输入：故事、角色、资产、模型配置、渲染任务
输出：可审核、可发布的剧集交付包
不可接受的失败：用户无法创建分镜、上传资产、预览媒体、生成演示数据、理解 QA/交付流程
```

### 1.2 核心业务闭环验证

必须识别系统最小闭环，并实际跑通。

格式：

```text
闭环名称：
步骤 1：
步骤 2：
步骤 3：
最终产物：
验证结果：通过/失败
失败点：
```

任何系统如果最小闭环跑不通，不得声称可用。

### 1.3 角色与场景验证

至少验证：

| 维度 | 必查问题 |
|------|----------|
| 用户角色 | 谁在用？管理员、运营、创作者、审核员、开发者分别做什么？ |
| 使用频率 | 高频路径是否最短？低频配置是否解释清楚？ |
| 前置条件 | 用户第一次进系统时是否知道先做什么？ |
| 失败恢复 | 用户操作失败后是否知道怎么继续？ |
| 交付物 | 用户最终能拿到什么？文件、链接、报告、任务状态？ |
| 多角色隔离 | 不同角色看到的数据和操作是否正确隔离？ |

### 1.4 边界场景识别

必须识别并验证系统边界：

| 场景类型 | 示例 |
|----------|------|
| 数据边界 | 空数据、单条数据、大量数据、超长文本、特殊字符、Unicode/emoji |
| 时间边界 | 跨天操作、时区切换、夏令时、并发时间戳 |
| 权限边界 | 未登录、过期 token、角色切换、越权访问 |
| 网络边界 | 断网、超时、弱网、重连 |
| 并发边界 | 多用户同时操作同一资源、多标签页同时操作 |
| 容量边界 | 大文件上传、大批量操作、高频请求 |

## 2. 架构与技术选型验证

### 2.1 前后端技术栈识别

测试前必须先确认：

```text
前端语言：
前端框架：
状态管理方式：
样式方案：
路由方案：
表单方案：
后端语言：
后端框架：
数据库：
ORM / migration：
文件存储：
缓存方案：
任务队列 / 异步任务：
实时通信（WebSocket/SSE/polling）：
搜索方案：
部署方式：
```

不知道技术栈就不能评价风险。

### 2.2 架构合理性检查

| 检查项 | 必查问题 |
|--------|----------|
| 分层 | 前端、后端、数据库、存储、任务队列边界是否清晰？ |
| 模块 | 模块是否按业务边界拆分，还是一堆页面/接口乱堆？ |
| 依赖 | 是否存在循环依赖、跨层直接调用、前端硬编码后端内部结构？ |
| 数据流 | 数据从输入到输出的路径是否清楚？是否有隐藏状态？ |
| 扩展性 | 新增一个业务对象或流程时是否需要全局大改？ |
| 可替换性 | 外部模型/API/存储是否可替换，还是写死？ |
| 可测试性 | 核心逻辑是否能独立测试，还是只能靠点页面？ |
| 关注点分离 | 业务逻辑是否泄漏到 UI 层？数据转换是否在正确层级？ |

### 2.3 抗压与稳定性初评

必须检查：

- 大列表是否分页/懒加载
- 上传大文件是否流式处理，而不是一次读入内存
- 长任务是否异步化，而不是阻塞 HTTP 请求
- 前端是否防重复提交
- 后端是否有超时、重试、幂等机制
- 数据库是否有索引
- 日志是否能定位一次失败请求
- 外部 API 失败是否会拖垮主流程
- 连接池大小是否合理，是否存在连接泄漏
- 是否有熔断/降级机制
- 后台任务失败是否有重试和死信队列
- 缓存策略是否合理，缓存失效是否正确

### 2.4 安全与漏洞初评

必须检查：

| 类型 | 必查问题 |
|------|----------|
| 鉴权 | 关键 API 是否需要登录/权限？是否裸奔？ |
| 越权 | 用户能否访问/修改不属于自己的数据？能否通过改 ID 遍历他人数据？ |
| 输入校验 | 表单/API 是否校验类型、长度、枚举、必填？ |
| 注入 | SQL/命令/路径是否有注入风险？ |
| 文件上传 | 是否限制文件类型、大小、路径穿越、可执行文件？ |
| XSS | 富文本/用户输入是否直接渲染 HTML？是否使用 dangerouslySetInnerHTML？ |
| Secret | API key 是否暴露到前端、日志、仓库？ |
| SSRF | 后端是否允许用户传 URL 后服务端抓取？ |
| CSRF | 是否有 CSRF token 保护？Cookie 是否设置 SameSite？ |
| 限流 | API 是否有限流？能否被暴力请求打垮？ |
| 信息泄露 | 错误响应是否暴露堆栈、内部路径、数据库结构？ |
| Token 安全 | Token 存储位置是否安全？过期机制是否合理？刷新机制是否正确？ |

### 2.5 可迭代性与编码质量

必须检查：

- 是否存在大量硬编码 URL、ID、状态、模型名、密钥名
- 是否存在 magic number / magic string
- 是否有统一 API client
- 是否有统一错误处理
- 是否有统一 loading/toast/modal 机制
- 是否有类型定义或 schema
- 是否有 migration，而不是手工改 DB
- 是否有明显重复代码
- 是否有过度抽象，导致简单需求难以改动
- 是否有临时 mock 混进真实流程且未标注
- 是否有"先写死，后面再说"的坑
- 环境配置是否与代码分离
- 是否有统一的日志格式和级别规范

## 3. 后端深度验证规则

### 3.1 API 面完整性

必须列出全部 API，并判断：

| 分类 | 检查项 |
|------|--------|
| 存在性 | 前端调用的接口后端是否真实存在？ |
| 僵尸接口 | 后端有接口但前端永远不用，是否应删除或文档化？ |
| 影子接口 | 前端调用了不存在/未文档化/被 rewrite 掩盖的接口？ |
| 404 接口 | 浏览器 Network 是否出现 404 API？ |
| 尾斜杠 | `/api/foo` 与 `/api/foo/` 是否一致？ |
| 方法 | GET/POST/PATCH/DELETE 是否与前端调用一致？ |
| 参数 | query/path/body 参数是否一致？ |
| 返回 | 返回字段是否与前端使用字段一致？ |
| 版本 | API 版本是否一致？是否有未版本化的破坏性变更？ |

### 3.2 API 契约验证

每个核心 API 必须验证：

```text
接口：
方法：
请求参数：
请求示例：
成功响应字段：
失败响应字段：
前端使用位置：
验证结果：
```

### 3.3 CRUD 真实性与数据完整性

对每个业务对象必须验证：

1. POST 创建后，GET 能查到
2. PATCH/PUT 更新后，GET 返回新值
3. DELETE 删除后，GET 不再返回
4. 空列表返回 `[]`，不是 404/500
5. 不存在 ID 返回 404，不是 500
6. 缺必填字段返回 422/400，不是 500
7. 创建关联对象时，主对象引用是否正确更新
8. 删除主对象时，关联对象是否正确处理（级联/拒绝/保留）
9. 部分更新（PATCH）是否只修改指定字段，不覆盖未传字段

#### 数值与类型安全

| 检查项 | 必查问题 |
|--------|----------|
| 数值精度 | 金额/比例等场景是否用整数或 decimal，而非 float？ |
| 整数溢出 | 计数器/ID 是否有溢出风险？ |
| null 语义 | null、空字符串、空数组、0、false 是否被正确区分和处理？ |
| 时区 | 所有时间字段是否统一 UTC 存储？前端展示是否转换到用户时区？ |
| 字符编码 | Unicode、emoji、特殊字符（引号/尖括号/null byte）是否正确处理？ |
| 字段长度 | 是否校验最大长度？超长输入是否被截断或拒绝？ |

### 3.4 并发与竞态

必须检查：

| 场景 | 必查问题 |
|------|----------|
| 乐观锁 | 并发更新同一资源时是否检测冲突？ |
| 悲观锁 | 长事务是否锁定必要资源？是否死锁？ |
| 唯一约束 | 并发创建同名/同键记录是否报唯一冲突而非静默覆盖？ |
| 幂等性 | 重复请求是否产生重复副作用？POST 是否支持幂等键？ |
| 排队 | 限流/排队机制是否生效？超限时返回什么？ |
| 资源争抢 | 限量资源（库存/配额/名额）并发分配是否准确？ |

### 3.5 批量操作

必须检查：

| 检查项 | 必查问题 |
|--------|----------|
| 批量创建 | 10/100/1000 条批量创建是否正常？是否有上限？ |
| 批量更新 | 部分成功部分失败时如何处理？是否回滚或返回失败明细？ |
| 批量删除 | 大量删除是否超时？是否异步化？ |
| 部分失败 | 批量操作中一条失败是否影响其他条？ |
| 性能 | 批量操作是否用事务批量提交，而非逐条执行？ |

### 3.6 分页与查询

必须检查：

| 检查项 | 必查问题 |
|--------|----------|
| 分页一致性 | 翻页时是否有新增/删除导致数据跳过或重复？ |
| 总数准确 | total 是否与实际数据量一致？ |
| 排序稳定 | 排序字段值相同时顺序是否稳定？ |
| 游标分页 | 若使用游标分页，游标是否正确编码解码？ |
| 搜索正确性 | 模糊搜索/全文搜索结果是否符合预期？特殊字符是否破坏查询？ |
| 筛选组合 | 多条件筛选组合是否正确取交集？空筛选是否返回全部？ |
| 无结果 | 搜索无结果时是否正确返回空列表而非错误？ |

### 3.7 数据库设计验证

| 检查项 | 必查问题 |
|--------|----------|
| 表结构 | 是否符合业务实体？有没有一张万能大表？ |
| 主键 | 主键是否稳定、唯一、不可变？是否使用自然键导致不可变性问题？ |
| 外键 | 关系是否明确？是否靠字符串散落关联？ |
| 索引 | 高频查询字段是否有索引？复合索引顺序是否合理？是否有冗余索引？ |
| 约束 | 唯一约束、非空约束、枚举约束、检查约束是否合理？ |
| migration | ORM 字段变更是否有 migration？是否靠手工 ALTER？ |
| 冗余 | 是否有不必要冗余字段？冗余是否有同步机制？ |
| JSON 字段 | JSON 是否被滥用来逃避建模？JSON 内字段是否需要索引？ |
| 软删 | 是否需要软删？软删后查询是否全局过滤？唯一约束是否排除已软删记录？ |
| 审计字段 | created_at / updated_at / created_by 是否完整？ |
| 数据隔离 | 多租户场景是否有 tenant_id 隔离？是否有全局查询遗漏租户条件？ |

### 3.8 ORM 与数据库一致性

必须检查：

- ORM 模型字段与真实表列一致
- 新增字段有 migration
- 默认值在 ORM 和 DB 侧一致
- nullable 设置一致
- 枚举值一致
- 关系加载不会触发 N+1 失控
- 软删 scope 是否在所有查询中生效
- 数据库表名/列名映射是否正确

### 3.9 后台任务与异步处理

必须检查：

| 检查项 | 必查问题 |
|--------|----------|
| 任务状态 | 异步任务是否有状态追踪（pending/running/done/failed）？ |
| 重试 | 失败任务是否自动重试？重试间隔和次数是否合理？ |
| 死信 | 多次重试仍失败是否进入死信队列？是否有告警？ |
| 超时 | 长任务是否有超时机制？超时后如何清理？ |
| 幂等 | 重试是否产生重复副作用？ |
| 前端感知 | 前端是否能查询任务进度？轮询/推送是否合理？ |
| 资源清理 | 失败任务是否清理临时文件/中间状态？ |

### 3.10 缓存与数据一致性

必须检查：

| 检查项 | 必查问题 |
|--------|----------|
| 缓存失效 | 数据更新后缓存是否正确失效？ |
| 过期策略 | 缓存过期时间是否合理？热点数据是否永不过期导致脏读？ |
| 穿透保护 | 查询不存在的数据是否每次都打到数据库？ |
| 击穿保护 | 热点 key 过期时是否大量请求同时打到数据库？ |
| 序列化 | 缓存序列化/反序列化是否正确？类型是否丢失？ |

### 3.11 后端可靠性

| 风险 | 检查方法 |
|------|----------|
| 500 泛滥 | 扫所有核心 API，确认没有未捕获异常 |
| 事务不完整 | 创建多张表时中途失败是否回滚？ |
| 长任务阻塞 | 渲染/生成/上传是否阻塞请求线程？ |
| 外部服务失败 | 模型/API/存储失败是否有降级和错误提示？ |
| 日志不足 | 失败时是否能从日志定位请求、用户、参数、堆栈？ |
| 资源泄漏 | 文件句柄、数据库连接、HTTP 连接是否正确释放？ |
| 优雅关闭 | 服务重启时是否处理完进行中的请求？ |

### 3.12 后端安全深度检查

必须至少做静态检查：

- 是否把 API key 写入代码/前端 bundle
- 文件上传路径是否可穿越
- 是否允许上传可执行文件并被访问
- 是否有未鉴权的敏感管理接口
- 是否把异常堆栈直接返回给前端
- 是否存在命令执行、任意文件读取、SSRF 风险
- 是否有越权访问（水平：同角色不同用户；垂直：低权限访问高权限接口）
- Token 是否有合理的过期和刷新机制
- 敏感数据（密码、手机号、身份证）是否脱敏存储和返回
- 是否有请求频率限制
- 文件下载是否校验权限

### 3.13 Webhook 与外部集成

如果系统发送或接收 webhook：

| 检查项 | 必查问题 |
|--------|----------|
| 签名验证 | 接收的 webhook 是否验证签名？ |
| 重试 | 发送的 webhook 失败是否重试？ |
| 幂等 | 接收端是否幂等处理重复通知？ |
| 超时 | webhook 处理是否超时？超时后发送端如何处理？ |
| 安全 | webhook URL 是否包含不可预测的 token？ |

## 4. 前端深度验证规则

### 4.1 功能优先级

前端验证顺序必须是：

1. 功能是否能完成目标
2. 接口是否真实存在且契约一致
3. URL 状态与导航是否正确
4. 状态是否正确（loading/empty/error/success/disabled）
5. 交互是否合理丝滑
6. 表单是否健壮
7. 样式是否统一
8. UI 是否有审美和产品感
9. 性能是否可接受
10. 可访问性是否达标

不能先看 UI 漂亮，再忽略功能不可用。

### 4.2 页面级验证

每个页面必须记录：

```text
页面 URL：
页面目标：
用户在此页面要完成什么：
依赖 API：
核心按钮/入口：
正常状态：
空状态：
错误状态：
加载状态：
权限受限状态：
验收结论：
```

### 4.3 接口完整性扫描

必须扫描前端所有接口调用：

- `fetch(`
- `axios.`
- `apiClient.`
- `useSWR` / `useQuery` / `react-query`
- 表单 action
- Next.js route handler / rewrite
- WebSocket 连接
- EventSource (SSE)

并生成：

| 前端调用 | 方法 | 后端是否存在 | 是否有 rewrite | 状态 |
|----------|------|--------------|----------------|------|

分类：

- **正常接口**：前端调用、后端存在、契约一致
- **影子接口**：前端调用但后端没有真实实现
- **僵尸接口**：后端存在但前端不用、文档也没写
- **404 接口**：浏览器实际请求返回 404
- **假接口**：返回 mock 成功但没有真实业务效果

影子/404 接口必须修复；僵尸接口必须解释保留原因或清理。

### 4.4 URL 状态与导航

必须检查：

| 检查项 | 必查问题 |
|--------|----------|
| 深链直达 | 通过 URL 直接打开子页面/详情页是否能正确加载？ |
| 刷新恢复 | 刷新页面后状态是否正确恢复（不跳首页、不丢数据）？ |
| 前进后退 | 浏览器前进/后退是否行为正确？ |
| 参数持久化 | 筛选/分页/搜索参数是否反映在 URL 中？ |
| URL 可分享 | 复制 URL 给他人打开是否看到相同状态？ |
| 无效路由 | 访问不存在的路由是否返回 404 页面？ |
| 重定向 | 登录前访问需认证页面是否正确重定向？ |

### 4.5 认证与权限 UI

必须检查：

| 检查项 | 必查问题 |
|--------|----------|
| 登录流程 | 输入错误凭据 → 提示错误 → 正确凭据 → 登录成功 → 跳转 |
| 登出流程 | 登出后是否清除状态？再访问需认证页面是否跳转登录？ |
| Token 过期 | Token 过期后操作是否触发重新登录或静默刷新？ |
| 权限隐藏 | 无权限的功能入口是否隐藏或禁用？ |
| 越权兜底 | 直接输入 URL 访问无权限页面，前端是否正确拦截？ |
| 角色切换 | 切换角色后 UI 是否正确更新？ |

### 4.6 前端功能验证

每个可操作功能必须实际验证：

| 功能 | 必测动作 |
|------|----------|
| 创建 | 打开表单 → 填写 → 提交 → 列表/详情更新 |
| 编辑 | 打开已有数据 → 修改 → 保存 → 数据更新 |
| 删除 | 删除 → 确认 → 数据消失 |
| 上传 | 选择文件 → 上传 → 进度显示 → 列表新增 → 可下载/预览 |
| 下载 | 点击下载 → 文件正确保存 → 文件内容完整 |
| 预览 | 点击预览 → 真实媒体/内容显示 |
| 搜索 | 输入关键词 → 结果变化 → 清空恢复 |
| 筛选 | 切换筛选项 → 结果变化 → 重置恢复 |
| 分页 | 下一页/上一页 → 数据变化 → 跳转指定页 |
| 选择 | 下拉/单选/多选 → 状态联动正确 |
| 跳转 | 点击链接 → 到正确页面 |
| 批量 | 多选 → 批量操作 → 结果正确 |

### 4.7 表单深度验证

必须检查：

| 检查项 | 必查问题 |
|--------|----------|
| 即时校验 | 输入时是否实时校验？错误提示是否清晰定位到字段？ |
| 提交校验 | 提交空表单/必填未填是否阻止提交？ |
| 脏状态 | 修改表单后未保存离开页面是否提示？ |
| 自动保存 | 长表单是否有自动保存？断网/关闭后恢复是否正确？ |
| 多步表单 | 步骤间数据是否保持？上一步修改是否反映到下一步？ |
| 重置 | 重置表单是否清空所有字段和状态？ |
| 联动 | 字段间联动（选 A 显示 B）是否正确？ |
| 浏览器自动填充 | 浏览器自动填充是否与表单交互冲突？ |
| 复制粘贴 | 从 Excel/富文本粘贴是否正确处理？ |

### 4.8 状态验证

每个页面必须验证 5 种状态：

1. **loading**：加载中是否有提示，是否会结束，骨架屏是否与真实内容结构匹配
2. **success**：成功后是否有反馈，数据是否更新，是否自动刷新相关列表
3. **empty**：无数据时是否告诉用户为什么空、下一步做什么，是否有引导入口
4. **error**：错误时是否显示可理解提示、是否可重试、网络错误与业务错误是否区分
5. **disabled**：按钮不可点时是否解释原因、是否提供前置操作入口

### 4.9 数据新鲜度与乐观更新

必须检查：

| 检查项 | 必查问题 |
|--------|----------|
| 乐观更新 | 操作后 UI 是否立即反映，失败后是否回滚？ |
| 列表刷新 | 创建/更新/删除后列表是否自动刷新？ |
| 轮询 | 长任务是否有合理的轮询间隔？页面切走后是否停止轮询？ |
| 缓存过期 | SWR/React Query 缓存时间是否合理？ |
| 多标签同步 | A 标签修改数据，B 标签是否感知？ |
| WebSocket | WebSocket 断连后是否自动重连？重连后状态是否同步？ |

### 4.10 交互合理性

必须检查：

- 按钮点击是否有反馈（hover/active/disabled 态）
- 提交中是否防重复点击
- 弹窗是否可关闭、ESC 是否有效、点击遮罩是否关闭
- 表单错误是否定位到具体字段并自动聚焦
- 操作成功后用户是否知道下一步
- 页面跳转是否卡顿、是否出现长时间白屏
- 多步骤流程是否有进度提示和可回退
- 危险操作是否二次确认，确认文案是否说明后果
- 空状态是否给下一步按钮
- 拖拽操作是否有正确的拖拽反馈和放置区域高亮
- 通知/toast 是否有正确的堆叠行为、自动消失、可手动关闭

### 4.11 错误边界与降级

必须检查：

- React Error Boundary 或等价机制是否兜住组件崩溃？
- 崩溃后是否显示友好的错误页面而非白屏？
- 图片/视频加载失败是否有占位符？
- 外部脚本/字体加载失败是否阻塞页面？
- 核心功能不可用是否有降级方案？

### 4.12 UI 与设计质量

不是只看能不能用，还要看是否像真正产品。

| 维度 | 必查问题 |
|------|----------|
| 一致性 | 按钮、卡片、表格、弹窗、输入框风格是否统一？ |
| 层级 | 标题、说明、操作按钮主次是否清楚？ |
| 配色 | 颜色是否统一，有无乱用渐变/荧光/高饱和？ |
| 字体 | 字号、行高、字重是否统一？ |
| 间距 | padding/margin 是否一致？是否拥挤或松散？ |
| 组件 | 是否复用组件，还是每页手写一套？ |
| 可读性 | 文案是否像人写的，还是 AI 套话？ |
| 产品感 | 页面是否解释"做什么/为什么/下一步"，还是堆按钮？ |
| 泛 AI 化 | 是否存在空洞大词、玻璃拟态滥用、无意义图标、假高级感？ |
| 死板感 | 是否只有表格+按钮，没有流程引导、状态反馈和上下文？ |
| 暗色模式 | 是否支持暗色模式？暗色模式下对比度是否足够？切换是否平滑？ |

### 4.13 响应式与可访问性

至少验证：

- 桌面 1440x900
- 平板 768x1024
- 移动端 375x812
- 侧边栏在小屏是否收起或抽屉化
- 弹窗是否超屏、是否可滚动
- 表格是否可横向滚动
- 按钮是否足够大（最小 44x44px 触控区域）
- 键盘 Tab 是否能访问主要控件，顺序是否合理
- 焦点样式是否可见
- 输入框是否有 label 或 aria-label
- 图标按钮是否有 title 或 aria-label
- 图片是否有 alt 文本
- 颜色对比度是否达到 WCAG AA（4.5:1）
- `prefers-reduced-motion` 是否禁用动画
- 屏幕阅读器是否能理解页面结构和操作

### 4.14 性能与资源管理

必须关注：

- 首屏是否明显卡顿（LCP < 2.5s 目标）
- 页面跳转是否超过 2 秒无反馈
- 大列表是否一次渲染过多数据（虚拟滚动？）
- 图片/视频是否懒加载
- 上传/预览大文件是否卡死页面
- 是否有重复请求、循环请求、请求风暴
- 浏览器 console 是否持续报错
- Bundle 体积是否合理？是否有不必要的全量引入？
- 代码分割是否正确（路由级别？组件级别？）
- 是否有内存泄漏（事件监听器、定时器、订阅未清理）
- 滚动位置在导航后是否恢复

## 5. 前后端联调验证

### 5.1 契约一致性

必须逐项确认：

| 项 | 检查 |
|---|------|
| URL | 前端请求 URL 与后端路由一致 |
| 方法 | GET/POST/PATCH/DELETE 一致 |
| 参数 | path/query/body 一致 |
| 字段 | 返回字段与前端读取字段一致 |
| 类型 | string/number/boolean/array/object 一致 |
| 空值 | null/undefined/空数组/空对象 处理一致 |
| 错误 | 错误响应格式前端能解析 |
| 认证 | token/cookie/header 一致 |
| 分页 | 分页参数和返回结构一致 |

### 5.2 浏览器 Network 验证

必须在真实浏览器或 Playwright 中检查：

- 是否有 4xx/5xx
- 是否有 pending 很久的请求
- 是否有重复请求
- 是否有 CORS 错误
- 是否有错误的 host/port
- 是否有 http/https 混用
- 是否有资源 403/404（图片、视频、字体、文件）
- 响应体大小是否异常（过大或过小）
- 请求头是否包含必要的认证信息

### 5.3 真实数据流验证

必须用真实数据跑：

```text
前端输入 → API 请求 → 后端处理 → 数据库写入 → 再次查询 → 前端展示
```

只验证 mock 数据不算真实联调。

### 5.4 认证全生命周期

必须端到端验证：

```text
注册/登录 → 获取 token → 请求带 token → 正常访问 → token 过期
→ 刷新 token 或重新登录 → 登出 → token 失效 → 重定向登录页
```

### 5.5 权限边界穿透测试

必须验证：

- 用户 A 能否通过 API 直接访问用户 B 的数据？
- 普通用户能否调用管理员接口？
- 未登录能否访问需认证的接口？
- Token 被篡改后是否被拒绝？

### 5.6 跨浏览器与环境

至少验证：

- Chrome（最新版）
- Firefox 或 Safari（至少一个）
- 移动端浏览器（Chrome iOS / Safari）

关注点：布局差异、CSS 兼容性、API 兼容性、触摸交互。

## 6. 项目级质量验证

### 6.1 可维护性

必须检查：

- 目录结构是否清楚
- 命名是否一致
- 类型定义是否集中
- API client 是否统一
- 错误处理是否统一
- UI 组件是否可复用
- 配置是否集中管理
- 测试脚本是否可重复运行
- 是否有 CI/CD 流水线
- 是否有环境隔离（dev/staging/prod）

### 6.2 过度设计与不合理设计

必须识别：

| 类型 | 表现 |
|------|------|
| 过度设计 | 简单 CRUD 搞复杂抽象、多层 wrapper、难以定位 |
| 欠设计 | 所有逻辑塞一个文件、状态散落、接口乱调 |
| 假设计 | 看起来有模块/服务，实际全是 mock 和空壳 |
| 不合理设计 | 数据模型不符合业务，导致功能绕来绕去 |
| 留坑设计 | 写死路径/ID/状态，后续一改就炸 |

### 6.3 文档与引导

系统必须说明：

- 怎么启动
- 怎么配置
- 需要哪些外部服务
- 哪些功能是真实能力
- 哪些功能是 mock/占位/说明入口
- 常见失败怎么排查
- 如何跑测试和回归
- API 文档是否存在且与实际一致

### 6.4 部署与可运维性

必须检查：

- 构建产物是否可复现
- 环境变量是否与代码分离
- 是否有健康检查端点
- 是否有日志聚合和查询能力
- 是否有监控和告警
- 是否有回滚机制
- 数据库迁移是否可在部署时自动执行
- 是否有数据备份策略

## 8. 可用等级

| 等级 | 定义 | 硬门禁 |
|------|------|--------|
| L0 | 代码能编译 | build/lint/type-check 通过 |
| L1 | 服务能启动 | 无启动错误，依赖服务可达 |
| L2 | 页面/API 可访问 | 核心路由 200，无启动崩溃 |
| L3 | 单页面核心功能可操作 | 至少一个页面的 CRUD 跑通 |
| L4 | 核心业务闭环跑通 | 最小闭环端到端真实跑通 |
| L5 | 架构/安全/体验/文档达到可交付标准 | 通过全部验证阶段 |

没有跑通核心业务闭环，不得超过 L3。
没有架构、安全、体验、文档评估，不得声称 L5。
有未修复的影子/404 接口，不得声称 L4+。

## 9. 禁止事项

1. 禁止把 build/tsc/HTTP 200 当成前端测试通过
2. 禁止不理解系统目标就开始验收
3. 禁止只测 happy path，不测空状态、错误状态、无数据状态
4. 禁止只看页面，不看 Network/API/数据库
5. 禁止只看功能，不看架构、可维护性、安全、性能
6. 禁止只看 UI 漂亮，不看业务是否跑通
7. 禁止把 mock 能跑当成真实能力可用
8. 禁止 worker 报告直接替代 main 复验
9. 禁止笼统说"已验证"，必须列出验证动作和证据
10. 禁止用户报告的问题不逐条回归
11. 禁止跳过并发/权限/数据完整性测试就声称"后端已验证"
12. 禁止不检查 URL 刷新/前进后退就声称"导航正常"
13. 禁止不检查 Token 过期/越权就声称"认证已验证"

## 10. 最低执行要求

如果用户说"全面测试 / 深度测试 / 前后端测试 / 可交付验收"，最低必须执行：

1. 系统目标与核心闭环梳理
2. 前端接口调用扫描，找影子/僵尸/404 接口
3. 后端 API CRUD 与错误处理验证
4. 并发与竞态场景检查
5. 分页与查询边界验证
6. ORM 与数据库结构一致性检查
7. 认证全生命周期端到端验证
8. 权限边界穿透测试
9. 前端 URL 状态与导航验证
10. Playwright 跑核心页面真实点击
11. 至少一个完整业务闭环真实跑通
12. 浏览器 Network 检查 4xx/5xx/资源错误
13. UI/交互/样式一致性检查
14. 架构/安全/可维护性风险清单
15. 按第 7 节格式输出验收报告
